Chapter 01 — Foundations
What is SQL?
Structured Query Language — the universal language for relational databases.
OverviewDefinition
SQL (Structured Query Language) is a standard language used to communicate with relational database management systems (RDBMS). It allows you to store, retrieve, update, and delete data. SQL is declarative — you describe what you want, not how to get it.
CategoryCommand groupCommands
DDLData Definition LanguageCREATE, ALTER, DROP, TRUNCATE
DMLData Manipulation LanguageINSERT, UPDATE, DELETE, SELECT
DCLData Control LanguageGRANT, REVOKE
TCLTransaction Control LanguageCOMMIT, ROLLBACK, SAVEPOINT
DatabasesPopular RDBMS
DatabaseUse caseNotes
MySQLWeb appsMost popular open-source DB
PostgreSQLComplex appsAdvanced features, very powerful
SQLiteLocal/mobileFile-based, no server needed
SQL ServerEnterprise (.NET)Microsoft ecosystem
OracleEnterpriseLarge-scale, high performance
StructureHow a relational database is organized
Database — top-level container (e.g., school_db)
Tables — like spreadsheets; each table stores one type of entity (e.g., students, courses)
Rows (Records) — each row is one data entry (e.g., one student)
Columns (Fields) — each column is an attribute (e.g., name, age, email)
Primary Key — uniquely identifies each row in a table
Foreign Key — links one table to another

Chapter 03 — Foundations
DDL — Define Structure
Data Definition Language commands create and modify the structure of your database.
CREATECreating database & table
CREATE DATABASE school_db;
USE school_db;

CREATE TABLE students (
  id         INT           PRIMARY KEY AUTO_INCREMENT,
  name       VARCHAR(100) NOT NULL,
  age        INT           CHECK (age >= 0),
  email      VARCHAR(150) UNIQUE,
  gender     ENUM('M','F','Other'),
  dept_id    INT,
  enrolled   DATE          DEFAULT (CURRENT_DATE),
  is_active  BOOLEAN       DEFAULT TRUE,
  FOREIGN KEY (dept_id) REFERENCES departments(id)
);
AUTO_INCREMENT auto-generates sequential IDs. NOT NULL prevents empty values. UNIQUE ensures no duplicates in that column.
ALTERModifying existing tables
-- Add a new column
ALTER TABLE students ADD phone VARCHAR(15);

-- Add column at a specific position
ALTER TABLE students ADD gpa DECIMAL(3,2) AFTER age;

-- Change column data type
ALTER TABLE students MODIFY phone VARCHAR(20);

-- Rename a column
ALTER TABLE students RENAME COLUMN phone TO mobile;

-- Drop a column
ALTER TABLE students DROP COLUMN mobile;

-- Add a constraint
ALTER TABLE students ADD CONSTRAINT chk_age CHECK (age < 100);
DROPRemoving objects
-- Drop table (all data lost!)
DROP TABLE students;

-- Drop if it exists (no error)
DROP TABLE IF EXISTS students;

-- Drop a database
DROP DATABASE school_db;
TRUNCATEClear all data
-- Delete all rows, keep structure
TRUNCATE TABLE students;

-- Resets AUTO_INCREMENT counter
-- Faster than DELETE for clearing
-- Cannot be rolled back
TRUNCATE cannot be rolled back in most databases. Use with caution.
Chapter 04 — Core Operations
SELECT — Querying Data
The most-used SQL command. Retrieves data from one or more tables.
Basic SELECTSelecting columns
-- All columns
SELECT * FROM students;

-- Specific columns
SELECT name, age, email FROM students;

-- Column aliases
SELECT
  name  AS student_name,
  age   AS student_age,
  email AS contact
FROM students;

-- Remove duplicates
SELECT DISTINCT dept_id FROM students;

-- Calculated columns
SELECT name, age, age + 5 AS age_in_5_years FROM students;
ExpressionsCASE — conditional column logic
SELECT
  name,
  age,
  CASE
    WHEN age < 18 THEN 'Minor'
    WHEN age BETWEEN 18 AND 25 THEN 'Young Adult'
    ELSE 'Adult'
  END AS age_group
FROM students;
Query OrderFull SELECT syntax & execution order
#ClauseExecutedPurpose
1FROMFirstWhich table(s)
2JOINAfter FROMCombine tables
3WHEREBefore GROUPFilter rows
4GROUP BYAfter WHEREGroup rows
5HAVINGAfter GROUPFilter groups
6SELECTNear endPick columns
7DISTINCTAfter SELECTRemove duplicates
8ORDER BYNear endSort results
9LIMIT/OFFSETLastRestrict rows
Writing order (SELECT…FROM…WHERE) ≠ execution order (FROM…WHERE…SELECT). This is why you can't use SELECT aliases in WHERE.

INSERT — Adding Data
DML command to add new rows to a table.
INSERTInserting rows
-- Insert one row (specify columns)
INSERT INTO students (name, age, email, dept_id)
VALUES ('Priya Sharma', 21, 'priya@mail.com', 2);

-- Insert multiple rows
INSERT INTO students (name, age, email, dept_id)
VALUES
  ('Rahul Mehta',  22, 'rahul@mail.com',  1),
  ('Anita Joshi',  20, 'anita@mail.com',  3),
  ('Kunal Patil',  23, 'kunal@mail.com',  2);

-- Insert from another table
INSERT INTO graduates (name, age)
SELECT name, age FROM students WHERE age >= 22;

-- Insert or ignore duplicates (MySQL)
INSERT IGNORE INTO students (name, email)
VALUES ('Test', 'priya@mail.com');

-- Insert or update on conflict (MySQL)
INSERT INTO students (id, name, age)
VALUES (1, 'Priya S', 22)
ON DUPLICATE KEY UPDATE age = 22;
Chapter 06 — Core Operations
UPDATE & DELETE
Modify or remove existing rows. Always use WHERE — or you'll affect every row.
UPDATEUpdating rows
-- Update one row
UPDATE students
SET age = 23, email = 'new@mail.com'
WHERE id = 1;

-- Update using a calculation
UPDATE employees
SET salary = salary * 1.10    -- 10% raise
WHERE dept_id = 3;

-- Update with CASE
UPDATE students
SET grade = CASE
  WHEN score >= 90 THEN 'A'
  WHEN score >= 75 THEN 'B'
  ELSE 'C'
END;
Never forget WHERE in UPDATE. Always test with a SELECT first using the same WHERE clause.
DELETEDeleting rows
-- Delete a specific row
DELETE FROM students WHERE id = 5;

-- Delete with multiple conditions
DELETE FROM students
WHERE age < 18 AND is_active = FALSE;

-- Delete using a subquery
DELETE FROM students
WHERE dept_id NOT IN (SELECT id FROM departments);

-- Delete all rows (keeps structure)
DELETE FROM students;      -- can be rolled back
TRUNCATE TABLE students;  -- cannot be rolled back
Chapter 07 — Core Operations
Filtering & Sorting
WHERE, LIKE, IN, BETWEEN, ORDER BY, LIMIT — controlling what data comes back.
WHEREComparison operators
SELECT * FROM students WHERE age = 21;       -- equal
SELECT * FROM students WHERE age != 21;      -- not equal (<> also works)
SELECT * FROM students WHERE age > 20;       -- greater than
SELECT * FROM students WHERE age <= 22;      -- less or equal
SELECT * FROM students WHERE age BETWEEN 18 AND 25;
SELECT * FROM students WHERE dept_id IN (1, 2, 3);
SELECT * FROM students WHERE email IS NULL;
SELECT * FROM students WHERE email IS NOT NULL;
LIKEPattern matching
-- % = any number of chars, _ = exactly one char
SELECT * FROM students WHERE name LIKE 'P%';      -- starts with P
SELECT * FROM students WHERE name LIKE '%a';      -- ends with a
SELECT * FROM students WHERE name LIKE '%iya%';   -- contains iya
SELECT * FROM students WHERE name LIKE '_____';   -- exactly 5 chars
SELECT * FROM students WHERE name NOT LIKE 'P%'; -- doesn't start with P
LogicAND, OR, NOT — combining conditions
SELECT * FROM students WHERE age > 20 AND dept_id = 2;
SELECT * FROM students WHERE dept_id = 1 OR dept_id = 3;
SELECT * FROM students WHERE NOT (age < 18);

-- Parentheses control precedence
SELECT * FROM students
WHERE (dept_id = 1 OR dept_id = 2) AND age > 20;
ORDER & LIMITSorting and pagination
-- Sort ascending (default)
SELECT * FROM students ORDER BY age ASC;

-- Sort descending
SELECT * FROM students ORDER BY age DESC;

-- Multi-column sort
SELECT * FROM students ORDER BY dept_id ASC, age DESC;

-- Top 5 records
SELECT * FROM students ORDER BY age DESC LIMIT 5;

-- Pagination: page 3 with 10 per page
SELECT * FROM students ORDER BY id LIMIT 10 OFFSET 20;
Chapter 08 — Intermediate
Joins
Combine data from multiple related tables using a common column.
INNER JOIN
Matching rows in both
LEFT JOIN
All left + matched right
RIGHT JOIN
All right + matched left
FULL OUTER JOIN
All rows from both
INNER JOINMost common join
-- Get students with their department name
SELECT
  s.id, s.name, s.age,
  d.dept_name
FROM students s
INNER JOIN departments d
  ON s.dept_id = d.id
WHERE d.dept_name = 'Engineering'
ORDER BY s.name;
Only returns rows where dept_id exists in both tables. Students with no department are excluded.
LEFT JOINKeep all left-table rows
-- All students, with dept if they have one
SELECT s.name, d.dept_name
FROM students s
LEFT JOIN departments d ON s.dept_id = d.id;

-- Students with NO department assigned
SELECT s.name
FROM students s
LEFT JOIN departments d ON s.dept_id = d.id
WHERE d.id IS NULL;
Multiple JOINsJoining 3+ tables
-- students → departments → courses → grades
SELECT
  s.name, d.dept_name,
  c.course_name, g.score
FROM students s
JOIN departments d  ON s.dept_id   = d.id
JOIN enrollments e  ON s.id        = e.student_id
JOIN courses c      ON e.course_id = c.id
JOIN grades g       ON g.enroll_id = e.id;
SELF JOINJoin a table to itself
-- employees: id, name, manager_id
SELECT
  e.name  AS employee,
  m.name  AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;
CROSS JOINEvery combination (cartesian product)
-- All possible student-course combos
SELECT s.name, c.course_name
FROM students s
CROSS JOIN courses c;
Returns rows_in_A × rows_in_B results. Use carefully — 100 × 100 = 10,000 rows!

Aggregate Functions
Functions that compute a single result from a set of rows.
FunctionsAll aggregate functions
SELECT COUNT(*)              FROM students;  -- all rows
SELECT COUNT(email)         FROM students;  -- non-null emails
SELECT COUNT(DISTINCT age)  FROM students;  -- distinct age values

SELECT SUM(salary)          FROM employees; -- total salary
SELECT AVG(age)             FROM students;  -- average age
SELECT MIN(age)             FROM students;  -- youngest
SELECT MAX(salary)         FROM employees; -- highest salary

-- Multiple aggregates at once
SELECT
  COUNT(*) AS total,
  AVG(age) AS avg_age,
  MIN(age) AS youngest,
  MAX(age) AS oldest
FROM students;
Chapter 10 — Intermediate
GROUP BY & HAVING
Group rows to apply aggregates per category, then filter those groups.
GROUP BYAggregate per group
-- Count students per department
SELECT dept_id, COUNT(*) AS total
FROM students
GROUP BY dept_id;

-- With JOIN for readable dept names
SELECT d.dept_name, COUNT(s.id) AS total,
       AVG(s.age) AS avg_age
FROM students s
JOIN departments d ON s.dept_id = d.id
GROUP BY d.dept_name
ORDER BY total DESC;
HAVINGFilter groups (not individual rows)
-- Departments with more than 10 students
SELECT dept_id, COUNT(*) AS total
FROM students
GROUP BY dept_id
HAVING total > 10;

-- WHERE vs HAVING together
SELECT dept_id, AVG(age) AS avg_age
FROM students
WHERE is_active = TRUE      -- filters rows first
GROUP BY dept_id
HAVING avg_age > 21;        -- then filters groups
WHERE filters individual rows (before grouping). HAVING filters entire groups (after grouping). You need GROUP BY for HAVING to make sense.

Chapter 12 — Intermediate
NULL Handling
NULL means "no value" or "unknown" — not zero, not empty string. It behaves differently.
NULL rulesHow NULL works
NULL is not a value — it's the absence of a value. Any arithmetic with NULL returns NULL. Any comparison with NULL returns NULL (not true/false). Always use IS NULL / IS NOT NULL, never = NULL.
-- WRONG — always returns 0 rows
SELECT * FROM students WHERE email = NULL;

-- CORRECT
SELECT * FROM students WHERE email IS NULL;
SELECT * FROM students WHERE email IS NOT NULL;
FunctionsNULL-safe functions
-- IFNULL: replace NULL with a default
SELECT name, IFNULL(email, 'No email') AS contact
FROM students;

-- COALESCE: return first non-NULL value
SELECT name, COALESCE(mobile, email, 'No contact') AS contact
FROM students;

-- NULLIF: return NULL if two values are equal
SELECT NULLIF(age, 0) FROM students;  -- NULL if age is 0

-- NULL in aggregates: COUNT(*) vs COUNT(col)
SELECT
  COUNT(*) AS total_rows,   -- counts all rows
  COUNT(email) AS has_email -- skips NULLs
FROM students;
Chapter 13 — Advanced
Subqueries & CTEs
Queries nested inside queries, and cleaner alternatives using WITH.
SubqueryIn WHERE clause
-- Students older than average age
SELECT name, age
FROM students
WHERE age > (SELECT AVG(age) FROM students);

-- Students in departments with > 5 people
SELECT name FROM students
WHERE dept_id IN (
  SELECT dept_id FROM students
  GROUP BY dept_id
  HAVING COUNT(*) > 5
);
SubqueryIn FROM clause (derived table)
-- Average age per dept, filtered
SELECT dept_id, avg_age
FROM (
  SELECT dept_id, AVG(age) AS avg_age
  FROM students
  GROUP BY dept_id
) AS dept_stats
WHERE avg_age > 21;
EXISTSEXISTS vs IN
-- EXISTS: true if subquery returns any row
SELECT name FROM students s
WHERE EXISTS (
  SELECT 1 FROM enrollments e
  WHERE e.student_id = s.id
);

-- NOT EXISTS: students with no enrollments
SELECT name FROM students s
WHERE NOT EXISTS (
  SELECT 1 FROM enrollments e
  WHERE e.student_id = s.id
);
EXISTS stops at the first match (faster). Use EXISTS for large datasets, IN for small lists.
CTECommon Table Expressions — WITH
-- Single CTE
WITH dept_summary AS (
  SELECT dept_id, COUNT(*) AS total, AVG(age) AS avg_age
  FROM students
  GROUP BY dept_id
)
SELECT d.dept_name, ds.total, ds.avg_age
FROM dept_summary ds
JOIN departments d ON ds.dept_id = d.id;

-- Multiple CTEs (chained)
WITH
active_students AS (
  SELECT * FROM students WHERE is_active = TRUE
),
dept_counts AS (
  SELECT dept_id, COUNT(*) AS cnt
  FROM active_students
  GROUP BY dept_id
)
SELECT * FROM dept_counts WHERE cnt > 3;
Recursive CTEHierarchical / tree data
WITH RECURSIVE org_tree AS (
  -- Base case: root nodes (no manager)
  SELECT id, name, manager_id, 0 AS depth
  FROM employees
  WHERE manager_id IS NULL

  UNION ALL

  -- Recursive case: children of current level
  SELECT e.id, e.name, e.manager_id, ot.depth + 1
  FROM employees e
  INNER JOIN org_tree ot ON e.manager_id = ot.id
)
SELECT depth, name FROM org_tree ORDER BY depth, name;
Chapter 14 — Advanced
Window Functions
Perform calculations across a "window" of rows related to the current row — without collapsing rows like GROUP BY does.
SyntaxWindow function structure
function_name() OVER (
  PARTITION BY column   -- define the window groups
  ORDER BY column        -- order within each window
  ROWS BETWEEN ...       -- optional: frame clause
)
RankingROW_NUMBER, RANK, DENSE_RANK, NTILE
SELECT
  name, dept_id, salary,
  ROW_NUMBER()  OVER (PARTITION BY dept_id ORDER BY salary DESC) AS row_num,
  RANK()        OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rnk,
  DENSE_RANK()  OVER (PARTITION BY dept_id ORDER BY salary DESC) AS drnk,
  NTILE(4)      OVER (PARTITION BY dept_id ORDER BY salary DESC) AS quartile
FROM employees;
FunctionTie behaviorExample (tied at 2nd)
ROW_NUMBERNo ties, arbitrary1, 2, 3, 4
RANKSkips numbers1, 2, 2, 4
DENSE_RANKNo skipping1, 2, 2, 3
NTILE(n)Divides into n buckets1, 1, 2, 2
ValueLAG, LEAD, FIRST_VALUE, LAST_VALUE
SELECT
  sale_date, amount,
  LAG(amount, 1)  OVER (ORDER BY sale_date) AS prev_amount,
  LEAD(amount, 1) OVER (ORDER BY sale_date) AS next_amount,
  amount - LAG(amount,1) OVER (ORDER BY sale_date) AS diff
FROM sales;
AggregateRunning totals & moving averages
SELECT
  sale_date, amount,
  SUM(amount) OVER (ORDER BY sale_date) AS running_total,
  AVG(amount) OVER (
    ORDER BY sale_date
    ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
  ) AS moving_avg_3
FROM sales;
PatternTop N per group (classic use case)
-- Top 3 highest-paid employees per department
SELECT * FROM (
  SELECT
    name, dept_id, salary,
    RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rnk
  FROM employees
) ranked
WHERE rnk <= 3;
Chapter 15 — Advanced
Views
A view is a saved SELECT query treated as a virtual table. It doesn't store data — it reruns the query each time.
CREATE VIEWCreating and using views
-- Create a view
CREATE VIEW active_students_view AS
SELECT
  s.id, s.name, s.age,
  d.dept_name
FROM students s
JOIN departments d ON s.dept_id = d.id
WHERE s.is_active = TRUE;

-- Use it like a regular table
SELECT * FROM active_students_view WHERE dept_name = 'CS';

-- Update or replace a view
CREATE OR REPLACE VIEW active_students_view AS
SELECT * FROM students WHERE is_active = TRUE;

-- Drop a view
DROP VIEW IF EXISTS active_students_view;
BenefitsWhen to use views
BenefitExplanation
SimplicityHide complex JOIN logic behind a simple name
SecurityGrant access to a view, not the underlying table
ReusabilityAvoid copy-pasting the same complex query
AbstractionChange underlying tables without breaking code
Views are read-only by default. Updatable views exist but have many restrictions — avoid modifying data through views.
Chapter 16 — Advanced
Stored Procedures & Functions
Reusable SQL blocks stored in the database, executed with a single call.
ProcedureCreating & calling procedures
DELIMITER //

CREATE PROCEDURE get_dept_students(
  IN  dept_name VARCHAR(50),
  OUT total     INT
)
BEGIN
  SELECT s.name, s.age
  FROM students s
  JOIN departments d ON s.dept_id = d.id
  WHERE d.dept_name = dept_name;

  SELECT COUNT(*) INTO total
  FROM students s
  JOIN departments d ON s.dept_id = d.id
  WHERE d.dept_name = dept_name;
END //

DELIMITER ;

-- Call it
CALL get_dept_students('Engineering', @count);
SELECT @count;
FunctionUser-defined functions (return a value)
DELIMITER //
CREATE FUNCTION get_age_group(age INT) RETURNS VARCHAR(20)
DETERMINISTIC
BEGIN
  DECLARE result VARCHAR(20);
  IF age < 18 THEN SET result = 'Minor';
  ELSEIF age < 26 THEN SET result = 'Young Adult';
  ELSE SET result = 'Adult';
  END IF;
  RETURN result;
END //
DELIMITER ;

-- Use it in a query
SELECT name, age, get_age_group(age) AS age_group
FROM students;
Chapter 17 — Advanced
Transactions & ACID
Group multiple SQL operations into one atomic unit — either all succeed or all fail.
ACIDACID properties
PropertyMeaningExample
AtomicityAll or nothingTransfer deducts AND credits — or neither
ConsistencyData stays validBalance can't go below 0
IsolationConcurrent txns don't interfereTwo users booking same seat
DurabilityCommitted data survives crashesBank transfer written to disk
TransactionsCOMMIT & ROLLBACK
START TRANSACTION;

-- Transfer ₹500 from account 1 to account 2
UPDATE accounts SET balance = balance - 500 WHERE id = 1;
UPDATE accounts SET balance = balance + 500 WHERE id = 2;

-- Verify (optional)
SELECT balance FROM accounts WHERE id IN (1, 2);

-- Save changes permanently
COMMIT;

-- OR undo everything if error
ROLLBACK;
SAVEPOINTPartial rollback
START TRANSACTION;

INSERT INTO orders (product, qty) VALUES ('Pen', 10);
SAVEPOINT after_order;

UPDATE inventory SET stock = stock - 10 WHERE product = 'Pen';

-- Only roll back to savepoint, keep order
ROLLBACK TO SAVEPOINT after_order;

COMMIT;

Indexes & Performance
Indexes are data structures that speed up lookups — like a book index vs reading every page.
IndexesTypes of indexes
-- Regular index (speeds up lookups)
CREATE INDEX idx_email ON students(email);

-- Unique index (no duplicate values)
CREATE UNIQUE INDEX idx_unique_email ON students(email);

-- Composite index (multi-column)
CREATE INDEX idx_dept_age ON students(dept_id, age);

DROP INDEX idx_email ON students;

Chapter 19 — Advanced
Normalization
The process of organizing a database to reduce redundancy and improve data integrity.
1NFFirst Normal Form — atomic values
Rules: Each column must hold a single (atomic) value. No repeating groups or arrays in a column. Each row must be unique.
-- BAD (not 1NF): courses stored as a list
-- | id | name  | courses          |
-- | 1  | Priya | Maths, Physics   |

-- GOOD (1NF): one value per cell
-- | student_id | course   |
-- | 1          | Maths    |
-- | 1          | Physics  |
2NFSecond Normal Form — no partial dependency
Must be in 1NF. Every non-key column must depend on the WHOLE primary key (not just part of it). Applies only when primary key is composite.
-- BAD (not 2NF): dept_name depends only on dept_id (part of PK)
-- PK: (student_id, dept_id) | dept_name (partial dep on dept_id)

-- GOOD (2NF): separate departments table
-- students: student_id, name, dept_id
-- departments: dept_id, dept_name
3NFThird Normal Form — no transitive dependency
Must be in 2NF. No non-key column should depend on another non-key column.
-- BAD: zip_code → city (non-key determines non-key)
-- students: id, name, zip_code, city

-- GOOD (3NF): separate zip table
-- students: id, name, zip_code
-- zip_codes: zip_code, city
SummaryNormal forms at a glance
NFRuleFixes
1NFAtomic values, no repeating groupsMulti-valued columns
2NFNo partial dependency on composite PKRedundant dept info in enrollments
3NFNo transitive dependencycity stored with zip in students
BCNFEvery determinant is a candidate keyEdge cases 3NF misses
In practice, normalize to 3NF. Denormalize selectively for read-heavy performance-critical tables (reporting, analytics).
Chapter 20 — Advanced
Constraints & Keys
Rules enforced by the database to maintain data accuracy and integrity.
ConstraintsAll constraint types
CREATE TABLE orders (
  id          INT           PRIMARY KEY AUTO_INCREMENT,  -- unique + not null
  student_id  INT           NOT NULL,                   -- no nulls
  product     VARCHAR(100) NOT NULL,
  qty         INT           DEFAULT 1,                  -- default value
  price       DECIMAL(8,2)  CHECK (price > 0),          -- validation
  status      ENUM('pending','shipped','done') DEFAULT 'pending',
  order_code  VARCHAR(20)  UNIQUE,                     -- no duplicates
  FOREIGN KEY (student_id)                               -- referential integrity
    REFERENCES students(id)
    ON DELETE CASCADE                                   -- auto-delete child rows
    ON UPDATE CASCADE                                   -- auto-update child FK
);
KeysTypes of keys
Key typeDescription
Primary KeyUniquely identifies each row. One per table. NOT NULL + UNIQUE.
Foreign KeyLinks to another table's primary key. Enforces referential integrity.
Unique KeyEnsures no duplicates. Can have NULLs (unlike PK). Multiple per table.
Composite KeyPrimary key made of 2+ columns together.
Candidate KeyAny column(s) that could be a primary key.
Surrogate KeyAuto-generated ID (like AUTO_INCREMENT) — no business meaning.
Natural KeyKey from real-world data (email, Aadhaar no.) — meaningful but less flexible.
ON DELETEForeign key actions
ActionEffect when parent row is deleted
CASCADEAlso deletes child rows automatically
SET NULLSets FK column in child to NULL
SET DEFAULTSets FK column to its default value
RESTRICTPrevents deletion if child rows exist
NO ACTIONSame as RESTRICT (default in most DBs)
Use CASCADE for dependent data (orders → order_items). Use RESTRICT for important relationships where accidental deletes would be catastrophic.
